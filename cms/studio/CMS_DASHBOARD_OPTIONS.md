# CMS Dashboard Options for Esencial

## What the editor should see

```text
Esencial CMS
├── Dashboard
│   ├── Att granska
│   ├── Senast ändrat
│   ├── Svenska / English coverage
│   └── Snabblänkar: nytt projekt, bilder, webbplatsinställningar
├── Struktur
│   ├── Svenska projekt
│   ├── English projects
│   ├── Tjänster och services
│   └── Webbplatsinställningar
└── Presentation
    └── Sidan som den ser ut, med redigering på plats
```

## Option A - Calm editorial dashboard

This is the current implementation. The dashboard prioritises editorial work:

- projects waiting for review
- recent drafts
- Sanity project information
- team/user information
- direct navigation to Swedish and English project lists

Best for: daily use by one or two editors who want a low-noise workspace.

## Option B - Editorial operations dashboard

Add a richer content-operations layer:

- publication calendar and scheduled content
- stale content list, for example projects not reviewed in 12 months
- missing-image and missing-SEO queues
- Swedish/English translation coverage
- image-rights renewal reminders
- release health and last successful frontend build

Best for: a studio with recurring publication and review responsibilities.

Implementation: custom dashboard widgets using GROQ queries, plus a GitHub Actions status widget and explicit fields for review dates, permissions, and owners.

## Option C - Visual editing workspace

Add Sanity Presentation Tool after the frontend has a preview data path:

- Studio and website side by side
- live draft changes in the page layout
- click-to-edit overlays from a visible heading or image to the matching CMS field
- desktop, tablet, and mobile preview sizes
- Swedish/English route switching
- preview links for reviewers

Best for: editors who think visually and need to know exactly where an image or paragraph lands before publishing.

Technical dependency: the frontend must fetch draft content from Sanity and implement the Visual Editing protocol. The current static HTML export cannot show unsaved CMS changes by itself.

## Recommended sequence

1. Use Option A immediately.
2. Add the missing-content and translation queues from Option B.
3. Connect the frontend to Sanity at build time and add a protected staging preview.
4. Enable Option C with the Presentation Tool.
5. Add automation only where it removes repeated editorial work; keep the public website's SEO markup generated by code.

## Design principles

- The first screen answers: what needs my attention?
- Use Swedish labels for the Swedish editorial team, with English content labels where they describe an English page.
- Show status as plain language, not internal database values.
- Keep SEO fields in a dedicated section with help text and length validation.
- Keep canonical URLs, JSON-LD, redirects, sitemap, and crawler rules developer-controlled.
- Prefer one strong image preview and useful text over a dense control panel.
