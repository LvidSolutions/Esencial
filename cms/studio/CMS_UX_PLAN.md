# Esencial CMS UX Plan

## Objective

Give non-technical editors one calm place to create, review, and publish Esencial content without editing code, URLs by accident, or search-engine markup.

## Editor experience

### 1. Clear starting point

The Studio navigation is organised by real work rather than database types:

- **To review**: items that need a second pair of eyes before publication.
- **Swedish projects** and **English projects**: language-specific lists for daily editing.
- **Services**: future service pages, kept separate until their content is approved.
- **Website settings**: one protected document for contact details and defaults.

### 2. Guided project editing

Every project editor follows the same order:

1. Basics: title, language, permanent URL, place, year, publication status.
2. Content: short introduction and longer project story.
3. Images: published images, descriptions, credits, and legacy image migration references.
4. Search and sharing: title and description used for search and social cards.

The UI uses field descriptions and character limits. Technical fields such as JSON-LD, canonical URL, alternate-language links, sitemap entries, and image dimensions are generated by the website build and are not editable in Studio.

### 3. Publishing safety

- New documents begin as drafts.
- A reviewer moves a complete item to **In review**.
- A project can only be marked **Published** when it has a summary, a slug, an image, image descriptions, and search fields.
- Permanent URLs are generated from titles but changing an existing URL is treated as a deliberate editorial decision that needs a redirect in the website layer.

### 4. Image and language policy

- Existing static-site image URLs are visible as migration references; they are not silently copied into Sanity.
- New published images require an image description and optional credit.
- Swedish and English records share a translation key but have their own approved copy and URL.

### 5. Operations

- Editors use the hosted Studio at `https://esencial-cms.sanity.studio/`.
- Sanity holds editorial data. GitHub remains the build and deployment source until the public site is switched to fetch published Sanity data.
- A future build integration will generate pages, structured data, language links, sitemap entries, and previews from the published CMS records.

## Implementation status

Implemented in this delivery:

- Editorial navigation and review queue
- Swedish/English project lists
- Grouped, Swedish-language project and service forms
- Required-field validation and publication safeguards
- Initial templates for new Swedish and English projects
- Single-document site settings
- Help text for content, images, URLs, and SEO fields

Deferred until the frontend uses Sanity as its build source:

- Live page preview
- Automatic redirects after a slug change
- Automatic upload of legacy images into the Sanity asset library
- GitHub-triggered production site rebuild after publishing
